# Open Review — GitHub Comment Template Spec

This document is the single source of truth for how Open Review renders its PR review comment on GitHub.  
When updating the visual design, change this spec first, then update `formatter.js` to match.

---

## Philosophy

- Markdown-only. No custom CSS, no inline styles, no HTML that GitHub might sanitize inconsistently (except where explicitly allowed below).
- Visual hierarchy is created through spacing, typography, icons, and tables — not colors.
- Every section should be scannable at a glance.

---

## Sections (in order)

### 1. Verdict Badge

A single `forthebadge.com` image that communicates the top-level recommendation.

| Verdict | Primary Label | Secondary Label | Primary BG |
|---|---|---|---|
| `approve` | LGTM | approve and merge | `#2ea44f` |
| `request_changes` | CHANGES REQUESTED | do not merge | `#d73a49` |
| `comment` | HOLD | let's discuss the approach | `#d4a017` |

- **Height**: `32px`
- **Alt text**: empty (decorative)

---

### 2. Top-Level Summary

Plain text paragraph shown **only** when the verdict is not `approve`.  
No callouts, no blockquotes, no alert syntax.

---

### 3. Summary Table

A bolded heading `**Review Overview**` followed by a full-width HTML table (`<table width="100%">`) displaying the count of findings per section.

- **Headers**: shields.io flat-square badges with the section title.
- **Values**: bold, centered text.
- **Dynamic columns**: Only sections with `count > 0` are included.

| Section | Badge Color |
|---|---|
| Must fix | `d73a49` |
| Should fix | `d4a017` |
| Questions | `0969da` |
| Suggestions | `57606a` |

> **Configurability**: This table may be toggled off via `show_summary_table` in `.open-review.yml` (default: `true`).

---

### 4. Must fix

Rendered only when `severity === 'critical'` findings exist.

#### Header

```markdown
## <img src="ICON_STOP" height="16" alt=""> Must fix <img align="right" src="COUNT_BADGE" height="20">
```

- **Left icon**: Octicon `stop-16.svg` at `16px`
- **Right count**: shields.io badge `-{count}` in section color, `20px` height, `align="right"`
- **List style**: Numbered (`1.`, `2.`, etc.)

#### Body per finding

- **Title**: bold
- **File path**: clickable link to `https://github.com/{owner}/{repo}/blob/{sha}/{file}#L{line}` when repo metadata is available; otherwise plain `` `file:line` `` code span.
- **Description**: regular paragraph
- **Suggested fix**: fenced code block with language detection:
  - If the fix contains diff markers (`+ `, `- `, `@@`) → ` ```diff `
  - Otherwise infer from file extension (`php`, `js`, `ts`, `py`, etc.)
  - Fallback → plain code block

#### Section summary

Optional paragraph rendered directly under the header, sourced from `result.sectionSummaries?.mustFix`.

---

### 5. Should fix

Rendered only when `severity === 'warning'` findings exist.

#### Header

```markdown
## <img src="ICON_DIFF_REMOVED" height="16" alt=""> Should fix <img align="right" src="COUNT_BADGE" height="20">
```

- **Left icon**: Octicon `diff-removed-16.svg` at `16px`
- **Right count**: same style as Must fix, amber color
- **List style**: Bulleted (`-`)

#### Body per finding

Same structure as Must fix.

---

### 6. Questions

Rendered only when `type === 'question'` findings exist.

#### Header

```markdown
## <img src="ICON_COMMENT_DISCUSSION" height="16" alt=""> Questions <img align="right" src="COUNT_BADGE" height="20">
```

- **Left icon**: Octicon `comment-discussion-16.svg` at `16px`
- **Right count**: blue color
- **List style**: Bulleted (`-`)

#### Body per finding

- **Title**: bold
- **Description**: regular paragraph
- No file path, no suggested fix block (questions are conversational).

---

### 7. Suggestions

Rendered only when `type === 'suggestion'` or `severity === 'info'` (and not a question) findings exist.

#### Header

```markdown
## <img src="ICON_LIGHT_BULB" height="16" alt=""> Suggestions <img align="right" src="COUNT_BADGE" height="20">
```

- **Left icon**: Octicon `light-bulb-16.svg` at `16px`
- **Right count**: gray color

#### Collapsible wrapper

The entire body is wrapped in:

```html
<details>
<summary>View suggestions</summary>
...
</details>
```

#### Body per finding

Same structure as Should fix.

---

### 8. Footer

A single horizontal rule (`---`) followed by:

```markdown
*Generated by Open Review* · **v{version}** · {timestamp}

<!-- open-review:meta
version: {version}
generated: {ISO8601}
-->
```

- **Timestamp**: Formatted via `Intl.DateTimeFormat` in the configured `TIMEZONE` (default `UTC`).
- **Metadata block**: HTML comment used for upsert detection. Must remain intact.

---

## Spacing Rules

- **No** `---` between sections. Rely on whitespace and `##` headers for separation.
- One blank line before and after the Summary Table.
- One blank line after each section header before its summary paragraph.
- Two blank lines between the last section and the footer `---`.
